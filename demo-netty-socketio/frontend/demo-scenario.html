Netty-SocketIO Demo Scenario

Kịch bản demo real-time notification embedded trong Spring Boot

OneSubs Admin · Socket.IO protocol · 2026-04-07

🛠️ Chuẩn bị trước khi demo

  • Spring Boot + Netty-SocketIO running: cd demo-netty-socketio && ./gradlew bootRun
  • Verify logs: [netty-socketio] started on 0.0.0.0:9092 + Tomcat started on port 8080
  • Tab 1 (Alice): mở frontend/index.html → User ID alice → Connect
  • Tab 2 (Bob): mở thêm tab → User ID bob → Connect
  • Tab 3 (Carol): mở thêm tab → User ID carol → Connect
  • Mở Postman / Git Bash / PowerShell sẵn để publish qua REST

📋 Kịch bản

  1. Publish — Phát message vào 1 room cụ thể
  2. Broadcast — Phát cùng message vào nhiều rooms
  3. Broadcast All — Phát cho tất cả clients
  4. Stats — Đếm clients & rooms đang active
  5. Isolation — Verify room-based filtering
  6. In-process Publish — Inject bean từ Spring service
  7. Handshake Auth — Reject connection thiếu userId
SCENARIO 01

Publish — Phát vào 1 room POST /api/publish

Mô phỏng: Đơn hàng mới → phát vào room user#alice. Chỉ Alice nhận.

Request

POST http://localhost:8080/api/publish Content-Type: application/json { "room": "user#alice", "event": "notification", "data": { "title": "🎉 Đơn hàng mới #ORD-001", "body": "Khách Nguyễn Văn A đặt gói Premium 12 tháng", "level": "success" } }

curl (Git Bash / Linux)

curl -X POST http://localhost:8080/api/publish \ -H "Content-Type: application/json" \ -d '{"room":"user#alice","event":"notification","data":{"title":"🎉 Đơn mới","body":"Khách A vừa đặt","level":"success"}}'

PowerShell (Windows)

Invoke-RestMethod -Uri http://localhost:8080/api/publish ` -Method Post ` -ContentType "application/json" ` -Body '{"room":"user#alice","event":"notification","data":{"title":"🎉","body":"Test"}}'

Kết quả mong đợi FE

✓ Chỉ Tab Alice hiện toast + entry mới. Tab Bob, Carol im lặng. Counter Alice +1.
SCENARIO 02

Broadcast — Phát vào nhiều rooms POST /api/broadcast

Kết quả mong đợi

✓ Tab Alice + Bob nhận cùng lúc. Tab Carol không nhận.
SCENARIO 03

Broadcast All — Phát cho tất cả POST /api/broadcast-all

Mô phỏng: Emergency alert — mọi client đang connect đều nhận, không filter room.

Request

POST http://localhost:8080/api/broadcast-all { "event": "notification", "data": { "title": "🚨 Hệ thống gặp sự cố", "body": "Vui lòng dừng thao tác", "level": "error" } }

Kết quả mong đợi

✓ Cả 3 tab Alice + Bob + Carol đều nhận toast đỏ.

Điểm nhấn Java

⚡ Dưới capo gọi server.getBroadcastOperations().sendEvent(...). 0ms overhead vì inject bean trực tiếp.

SCENARIO 05

Isolation — Room-based filtering

Chứng minh: phát vào room X → chỉ client trong room X nhận. Đây là core value của room-based model.

Các bước

  1. Publish vào user#alice → chỉ Alice nhận
  2. Publish vào user#bob → chỉ Bob nhận
  3. Publish vào user#ghost (không ai subscribe) → không ai nhận, không error

Điểm nhấn

⚡ Message phát vào room trống → mất luôn. Netty-SocketIO không có history buffer như Centrifugo. Muốn persist phải lưu DB riêng.
SCENARIO 06

In-process Publish — Inject bean trong Spring service Java

Lợi thế lớn nhất so với Centrifugo: Không cần HTTP call, inject NotificationPublisher thẳng vào service domain.

Code example

// OrderService.java @Service @RequiredArgsConstructor public class OrderService { private final NotificationPublisher publisher; // inject trực tiếp private final OrderRepository orderRepo; public void createOrder(CreateOrderRequest req) { Order order = orderRepo.save(new Order(req)); // Cùng transaction context với domain event, 0 HTTP overhead publisher.publishToRoom( "user#" + order.getAssignedSaleId(), "notification", Map.of( "title", "Đơn mới #" + order.getId(), "body", order.getCustomerName(), "level", "success" ) ); } }

So sánh với Centrifugo

Aspect Centrifugo Netty-SocketIO
Publish call HTTP POST → :8000 Method call in-process
Latency ~1-5ms ~0ms
Transaction context Tách rời Cùng thread + tx
Debug Logs 2 services Breakpoint IDE
SCENARIO 07

Handshake Auth — Reject thiếu userId

Chứng minh: AuthorizationListener từ chối connection không có userId query param.

Test negative case

  1. Mở tab mới index.html
  2. Xóa trống field User ID
  3. Click Connect

Kết quả mong đợi

✓ FE: status hiện "Error: ..." / Disconnected ngay
✓ Server log: [handshake] rejected — missing userId

Code reference SocketIoConfig.java

cfg.setAuthorizationListener(data -> { String userId = data.getSingleUrlParam("userId"); if (userId == null || userId.isBlank()) { return AuthorizationResult.FAILED_AUTHORIZATION; } return AuthorizationResult.SUCCESSFUL_AUTHORIZATION; });

Upgrade path → JWT

💡 Production: thay query param bằng JWT. Parse data.getSingleUrlParam("token"), verify HS256 với shared secret, extract sub → làm userId.

🎯 Tổng kết demo

Những gì đã show

  1. In-process integration — inject SocketIOServer bean, 0ms overhead
  2. Room-based filtering — publish chính xác đến 1 hoặc nhiều rooms
  3. Broadcast all — emergency alert tất cả clients
  4. REST wrapper — Spring controller expose `/api/publish` cho external test
  5. Handshake auth — reject connection không hợp lệ ở tầng WS
  6. Cùng stack Java — debug/log/profile chung với Spring Boot

Ưu điểm so với Centrifugo

✅ Zero ops overhead — 1 JAR, không thêm container
✅ Cùng transaction context — publish sau khi DB commit thành công
✅ Socket.IO ecosystem — client library ~62K stars, đa platform (web/iOS/Android)
✅ Debug dễ — breakpoint IDE bắt được luồng từ controller → publisher → emit

Nhược điểm so với Centrifugo

❌ Không có Admin UI — phải self-build dashboard
❌ Không có presence/history built-in — tự implement với Redis hoặc DB
❌ Port riêng (9092) — FE phải config 2 endpoints (HTTP 8080 + WS 9092)
❌ Scale dính với app — không tách independent scaling như Centrifugo
