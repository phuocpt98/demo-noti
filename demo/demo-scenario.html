Centrifugo Demo Scenario

Kịch bản demo real-time notification qua Admin UI

OneSubs Admin · 2026-04-07

🛠️ Chuẩn bị trước khi demo

  • Centrifugo container running: docker compose up -d
  • Mở tab 1: Admin UI → http://localhost:8000 (login: admin / admin)
  • Mở tab 2: FE Receiver → demo/index.html → paste JWT → Connect (status "Connected")
  • (Tùy chọn) Mở thêm tab 3 với User ID khác để demo multi-user
  • Zoom browser lên ~125% để mọi người xem dễ

  1. Publish — Phát message vào 1 channel
  2. Broadcast — Phát cùng message vào nhiều channels
  3. Presence — Xem ai đang online trong channel

SCENARIO 01

Publish — Phát message vào 1 channel publish

Mô phỏng: Có đơn hàng mới → phát vào channel notifications:user#123. Ai đang subscribe channel đó sẽ nhận.

Các bước trên Admin UI

  1. Vào tab Actions → dropdown method chọn publish
  2. Điền channel: notifications:user#123
  3. Điền data:
{ "title": "🎉 Đơn hàng mới #ORD-001", "body": "Khách Nguyễn Văn A vừa đặt gói Premium 12 tháng", "level": "success" }
  1. Click Submit

Kết quả mong đợi FE

✓ Tab FE Receiver hiện ngay toast popup màu xanh + entry mới trong list "Received Notifications". Counter tăng lên 1.

Điểm nhấn khi trình bày

⚡ Nhấn mạnh: latency <50ms — click Submit → noti xuất hiện gần như tức thời. Không polling, không refresh.
SCENARIO 02

Broadcast — Phát cùng message vào nhiều channels broadcast

Mô phỏng: Admin thông báo maintenance — 1 API call phát vào N channels cùng lúc, ai subscribe bất kỳ channel nào trong list sẽ nhận.

Các bước trên Admin UI

  1. Actions → chọn method broadcast
  2. Điền channels (array):
[ "notifications:user#123", "notifications:user#456", "notifications:user#789" ]
  1. Điền data:
{ "title": "⚠️ Bảo trì hệ thống", "body": "Hệ thống sẽ tạm dừng 22:00 - 22:30 đêm nay", "level": "warning" }
  1. Submit

Kết quả mong đợi

✓ Nếu có mở nhiều tab FE với user id khác nhau → tất cả nhận cùng lúc. 1 API call → N clients.
SCENARIO 03

Presence — Ai đang online? presence_stats

Mô phỏng: Đếm số sale đang xem đơn hàng realtime — tính năng "● X người đang xem".

Các bước

  1. Actions → chọn presence_stats
  2. Điền channel: notifications:user#123
  3. Submit

Kết quả mong đợi Admin UI

✓ Response trả về:
{ "num_clients": 1, "num_users": 1 }

Test nâng cao

  1. Đóng tab FE Receiver → Submit lại presence_stats
  2. Giờ thấy num_clients: 0 — update realtime theo connection thật
SCENARIO 04

History — Load lại messages đã miss history

Mô phỏng: User F5 trang, cần load lại 10 noti gần nhất mà không cần query DB.

Các bước

  1. Publish vài message (lặp Scenario 01 ~3 lần với nội dung khác nhau)
  2. Actions → chọn history
  3. Điền:
{ "channel": "notifications:user#123", "limit": 10 }
  1. Submit

Kết quả mong đợi

✓ Response trả về array các publications gần nhất đã gửi vào channel đó, kèm offset + epoch.

Điểm nhấn

⚡ History là buffer trong Centrifugo (config: history_size: 10, history_ttl: 300s). Không cần DB. Hỗ trợ reconnect tự động load lại missed messages.
SCENARIO 05

Channels — Dashboard channels active channels

Mô phỏng: Admin xem tất cả channels đang có subscriber — overview traffic.

Các bước

  1. Actions → chọn channels
  2. Điền pattern: notifications:*
  3. Submit

Kết quả mong đợi Admin UI

✓ List channels + số client per channel. Có thể click vào tab Channels ở sidebar để xem giao diện dashboard đẹp hơn.
SCENARIO 06

Disconnect — Force kick user disconnect

Mô phỏng: User bị disable account → cần kick ra khỏi tất cả connection ngay lập tức.

Các bước

  1. Đảm bảo FE Receiver đang Connected
  2. Actions → chọn disconnect
  3. Điền:
{ "user": "user-123" }
  1. Submit

Kết quả mong đợi FE

✓ Tab FE Receiver → status dot chuyển về xám, text "Disconnected: ...". Console log event disconnected từ server.
SCENARIO 07

Info — Server health & stats info

Mô phỏng: DevOps monitor số connection, uptime, version.

Các bước

  1. Actions → chọn info
  2. Body để trống {}
  3. Submit

Kết quả mong đợi

✓ Response trả về nodes[] với: version, num_clients, num_users, num_channels, uptime. Dùng cho Prometheus/Grafana scraping.

DEEP DIVE — ACL

🔐 Phân quyền Channel — Logic quyết định

Trả lời: "1 JWT có user/role/channel — Centrifugo check như nào? Làm sao biết channel phân quyền kiểu gì?"

Nguyên tắc cốt lõi

⚡ Centrifugo KHÔNG có concept "role" built-in.
Nó chỉ biết 2 thứ: User identity (JWT.sub) + Channel naming rules.
"Phân quyền theo role" = convention đặt tên channel + BE logic.

Flow check khi client subscribe

Ví dụ: Client có JWT { sub: "alice" } gọi subscribe("chat:user#alice"). Centrifugo chạy 4 bước theo thứ tự:

1 Parse namespace từ tên channel
Tách theo dấu : đầu tiên.
"chat:user#alice" → namespace = chat, phần còn lại = user#alice
2 Namespace có tồn tại trong config.json?
❌ Không → REJECT "unknown channel" (error code 102)
✅ Có → load rules của namespace (presence, history, allow_subscribe_for_client, proxy_subscribe...)
3 Phân tích ký tự đặc biệt trong phần còn lại
Scan tìm # và ,:
  • • "user#alice" → user-limited (chỉ alice)
  • • "team#alice,bob,carol" → multi-user (một trong 3)
  • • "general" (không có) → public trong namespace
4 Áp quy tắc ACL theo priority
Centrifugo check theo thứ tự:
  1. proxy_subscribe = true? → POST đến BE endpoint → BE trả allow/deny
  2. allow_subscribe_for_client = false? → yêu cầu Sub-token riêng (ký từ BE)
  3. Có #userId? → verify JWT.sub == userId
  4. Có #a,b,c? → verify JWT.sub ∈ {a,b,c}
  5. Còn lại → ALLOW (public trong namespace)
💡 Kết: Centrifugo quyết định ACL dựa trên (1) tên channel + (2) namespace config — hoàn toàn không đọc field role trong JWT.

Bảng ví dụ check (JWT sub="alice")

Channel Centrifugo check gì? Kết quả
chat:general Namespace chat cho phép subscribe? ✅ Ai cũng vào
chat:user#alice JWT.sub == "alice"? ✅ OK
chat:user#bob JWT.sub == "bob"? ❌ REJECT
chat:team#alice,bob JWT.sub ∈ {alice, bob}? ✅ OK
chat:team#bob,carol JWT.sub ∈ {bob, carol}? ❌ REJECT
orders:456 Namespace có proxy_subscribe=true → Centrifugo gọi BE ⚙️ BE quyết

🔑 3 Cơ chế ACL

Cơ chế A — Native Channel Prefix (đơn giản nhất)

// user-limited: chỉ user có JWT.sub == alice "chat:user#alice" // multi-user: chỉ alice HOẶC bob "chat:team#alice,bob"
✅ Không cần config gì thêm — Centrifugo tự verify theo JWT.sub.
📍 Dùng cho: Personal notification, private chat, owner-based.

Cơ chế B — Subscribe Token (BE ký riêng cho channel)

// config.json { "namespaces": [{ "name": "notifications", "allow_subscribe_for_client": false }] }
// Flow 1. FE subscribe "notifications:role-admin" 2. Centrifugo: "Phải có sub-token" 3. FE → BE: "Cho tôi sub-token" 4. BE check DB: alice.role == "admin"? → ký token { sub: "alice", channel: "notifications:role-admin" } 5. FE gửi sub-token → Centrifugo verify → OK
✅ BE là người quyết định ACL — safe cho role-based.
📍 Dùng cho: Role-based broadcast, tenant-scoped channel.

Cơ chế C — Proxy Subscribe (BE quyết mỗi lần)

// config.json { "proxy": { "subscribe_endpoint": "http://backend:8080/centrifugo/subscribe" }, "namespaces": [{ "name": "orders", "proxy_subscribe": true }] }
// Flow 1. FE subscribe "orders:456" 2. Centrifugo POST → BE /centrifugo/subscribe { user: "alice", channel: "orders:456" } 3. BE query DB: alice là owner/assignee của order 456? 4. BE trả { allow: true/false } 5. Centrifugo allow/reject
✅ Logic tập trung BE, dễ audit, update realtime theo DB.
📍 Dùng cho: Complex ACL — owner, assignee, dynamic permission.

🎯 Khuyến nghị OneSubs — Convention đặt tên

Loại noti Channel pattern Cơ chế
Personal (đơn mới của sale) notifications:user#<id> A — Native #
Role-based (admin broadcast) notifications:role-admin B — Sub-token
Chat nhóm cố định chat:order#u1,u2,u3 A — Native ,
Per-order (owner/assignee động) orders:<id> C — Proxy subscribe

💡 Trả lời thẳng 2 câu hỏi

Q1: "1 token có user/role/channel thì check như nào?"
A: JWT connection token chỉ cần sub (user id). Không nhét role vào JWT để Centrifugo check — vì nó không đọc role. Centrifugo chỉ so JWT.sub với ký tự đặc biệt trong tên channel. Role check = BE tự làm khi issue sub-token (cơ chế B) hoặc trong proxy endpoint (cơ chế C).
Q2: "Làm sao biết channel đó phân quyền kiểu nào?"
A: Từ 2 nơi:
  1. Tên channel — có #userId, có , → Centrifugo tự biết (cơ chế A)
  2. Namespace config — allow_subscribe_for_client, proxy_subscribe → bắt buộc qua BE không
