Kết luận: Library hay Microservice?

→ Library (trong monolith)

OneSubs Admin là monolith, ít user (Admin/Sale/Tech ≈ vài chục connections). Tách microservice tạo complexity không cần thiết. Khi nào scale thực sự cần thì refactor library → microservice không khó.

Kiến trúc tích hợp

Netty-SocketIO (embed trong Spring Boot): FE (React) ←── Socket.IO ──→ Netty-SocketIO (port 9092) │ Cùng JVM process với Spring Boot (port 8008) │ Spring Boot inject SocketIOServer bean │ server.getRoomOperations("order-123") .sendEvent("notification", payload); Netty-SocketIO (standalone microservice): FE (React) ←── Socket.IO ──→ Netty-SocketIO JAR (port 9092, process riêng) ↑ Redis Pub/Sub │ Spring Boot (port 8008) │ redisTemplate.publish("ws:notify", payload) Centrifugo / Soketi: FE (React) ←── WebSocket ──→ Centrifugo/Soketi (port 8001) ↑ HTTP API / gRPC │ Spring Boot (port 8008) │ POST /api/publish { channel: "user#123", data: {...} } Mercure: FE (React) ←── SSE ──→ Mercure Hub (port 3000) ↑ HTTP POST + JWT │ Spring Boot (port 8008) │ POST /.well-known/mercure topic=/notifications/user/123 data={...} Authorization: Bearer <publisher-jwt>

Netty-SocketIO — Lợi thế Java cùng stack

  • Cùng ngôn ngữ Java — debug, logging, profiling chung
  • Có thể embed trong cùng JAR (không cần container riêng)
  • Socket.IO ecosystem lớn nhất (~62K stars client)
  • Namespace + Room native — tách concern dễ dàng
  • Gradle dependency đơn giản, không cần Docker

Netty-SocketIO — Nhược điểm

  • Chạy port riêng — FE phải connect 2 endpoint
  • Không tích hợp Spring Security — tự handle auth
  • Không có presence, history, admin UI built-in
  • Maintenance moderate — không active như Centrifugo
  • Embed cùng JVM = không scale WS độc lập

Centrifugo — Best overall cho microservice

  • Feature đầy đủ nhất: presence, history, recovery
  • Performance cao nhất (Go, 100K+ connections)
  • Admin dashboard built-in
  • Two-way messaging cho chat tương lai
  • Docker image nhẹ (15MB), actively maintained

Nhược điểm chung Microservice

  • Thêm container/process → ops overhead
  • Network latency giữa Spring Boot → WS server
  • Auth phải proxy qua BE hoặc shared JWT secret
  • Domain events cần HTTP callback thay vì in-process
  • CI/CD pipeline phức tạp hơn

Kết luận: Open-source nào cho Microservice?

→ Centrifugo (best overall) · Netty-SocketIO (best Java option)

Centrifugo thắng về features + scale + maintenance. Netty-SocketIO là lựa chọn tốt nhất nếu ưu tiên cùng stack Java — embed được trong cùng JAR, debug/profile chung, nhưng thiếu presence/history/admin UI. Với OneSubs hiện tại, microservice vẫn là overkill.

Roadmap đề xuất

Phase 1 (hiện tại — OS-125): realtime-library + Spring WebSocket STOMP → Push notification (đơn hàng mới, trạng thái, assignment) → Role-based topics (Admin/Sale/Tech) Phase 2 (tương lai): Mở rộng realtime-library → Live chat trên đơn hàng (two-way messaging) → Presence (online status) Phase 3 (khi cần scale): Tách realtime-library → Centrifugo microservice → Khi concurrent connections >500 → Khi cần horizontal scaling riêng cho WS

5. Mở rộng Chat — Khi nào cần tách Microservice?

Phân tích theo loại chat để quyết định kiến trúc phù hợp.

Tiêu chí Chat nội bộ (nhân viên ↔ nhân viên) Chat với khách hàng (end-user facing)
Mô tả Admin/Sale/Tech trao đổi trên đơn hàng Khách hàng chat trực tiếp với support
Số user đồng thời ~20-50 (nhân viên nội bộ) Hàng trăm → hàng nghìn
Message volume Thấp — chat theo đơn hàng, không liên tục Cao — nhiều conversation song song, liên tục
Availability yêu cầu Chấp nhận downtime khi deploy Always-on — mất kết nối = mất khách
Deploy impact Restart → 20 nhân viên reconnect nhanh, không ai để ý Restart → 500 khách đang chat bị ngắt → UX tệ
Scale pattern Scale cùng API server là đủ WS connections ≠ API load → cần scale riêng
Kiến trúc Giữ trong monolith — realtime-library Tách microservice — Centrifugo hoặc Netty-SocketIO

Decision Tree

Cần real-time BE → FE? │ ├── Notification only (one-way) │ └── realtime-library (Spring WS STOMP) trong monolith │ ├── + Chat nội bộ (nhân viên ↔ nhân viên) │ │ Concurrent: ~20-50 | Volume: thấp │ └── Mở rộng realtime-library — vẫn trong monolith │ └── + Chat với khách hàng (end-user facing) │ Concurrent: hàng trăm+ | Volume: cao | Uptime: critical │ ├── Ưu tiên cùng stack Java? │ └── Netty-SocketIO microservice (standalone JAR) │ └── Ưu tiên features + scale? └── Centrifugo microservice (Docker)

Chat nội bộ → Monolith

  • Ít user → Spring WS STOMP thừa sức
  • Cùng transaction context với domain events
  • Cùng auth (Spring Security), không proxy
  • Zero infra mới, zero ops overhead
  • Đúng roadmap Phase 2: "Trò chuyện trực tiếp"

Chat khách hàng → Microservice

  • Scale WS server độc lập khi traffic tăng
  • Deploy WS server không ảnh hưởng API (zero-downtime)
  • Centrifugo: presence, history, recovery có sẵn
  • Horizontal scaling dễ dàng với Redis adapter
  • Chỉ tách khi thực sự có nhu cầu (YAGNI)

Kết luận: Chat có cần tách Microservice?

→ Chat nội bộ: KHÔNG — giữ monolith

→ Chat khách hàng: CÓ — tách khi có nhu cầu

Hiện tại OneSubs chỉ cần notification + chat nội bộ (Phase 2) → monolith đủ. Nếu tương lai mở rộng ra chat với khách hàng (end-user), lúc đó mới tách realtime-library → Centrifugo microservice. Kiến trúc library hiện tại cho phép migration dễ dàng vì đã tách concern qua port/adapter.
