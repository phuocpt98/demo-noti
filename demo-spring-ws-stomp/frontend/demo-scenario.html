SCENARIO 01

Publish User — Personal destination POST /api/publish-user

Mô phỏng: Phát vào personal queue của user alice. Spring dùng SimpMessagingTemplate.convertAndSendToUser() để route theo Principal.

Request

POST http://localhost:8080/api/publish-user { "username": "alice", "data": { "title": "🎯 Đơn hàng mới", "body": "Chỉ Alice nhận — personal queue", "level": "success" } }

Kết quả mong đợi FE

✓ Chỉ Tab Alice hiện toast với destination /user/queue/notifications. Bob, Carol im lặng.

Cơ chế dưới capo

// NotificationPublisher.java messaging.convertAndSendToUser( "alice", "/queue/notifications", payload ); // Spring resolves "alice" to active session(s) via Principal // → routes message to /user/queue/notifications-user<sessionId>
SCENARIO 02

Publish Topic — Role-based broadcast POST /api/publish-topic

Mô phỏng: Gửi thông báo cho tất cả admin. Alice + Bob đang sub /topic/role/admin → cả 2 nhận. Carol sub role/sale → không nhận.

Request

POST http://localhost:8080/api/publish-topic { "topic": "role/admin", "data": { "title": "⚠️ Admin alert", "body": "Tất cả admin cần check dashboard", "level": "warning" } }

Kết quả mong đợi

✓ Tab Alice + Bob đều nhận (cùng sub /topic/role/admin).
✓ Tab Carol không nhận (đang sub /topic/role/sale).

Test thêm

  1. Đổi body: "topic": "role/sale" → Submit
  2. Chỉ Carol nhận, Alice + Bob im lặng
SCENARIO 03

Broadcast All — Global topic POST /api/broadcast-all

Emergency alert gửi vào /topic/broadcast. Tất cả clients đã sub sẽ nhận.

Request

POST http://localhost:8080/api/broadcast-all { "data": { "title": "🚨 Hệ thống gặp sự cố", "body": "Vui lòng dừng thao tác", "level": "error" } }

Kết quả mong đợi

✓ Tất cả 3 tab (Alice, Bob, Carol) đều nhận — destination hiển thị /topic/broadcast.
SCENARIO 04

Stats — SimpUserRegistry GET /api/stats

Spring tự track connected users qua SimpUserRegistry. Lấy stats không cần code thêm.

Request

GET http://localhost:8080/api/stats

Kết quả mong đợi

{ "totalUsers": 3, "usernames": ["alice", "bob", "carol"] }

Test động

  1. Đóng tab Carol → GET lại → totalUsers: 2
  2. Mở lại tab Carol → Connect → totalUsers: 3
  3. Registry tự update realtime qua session events của Spring.
SCENARIO 05

Multi-tab isolation — Principal-based routing

Chứng minh: Spring route personal message theo Principal chứ không phải sessionId thô.

Setup

  1. Mở thêm 1 tab nữa với cùng username alice → Tab 1b
  2. Bây giờ alice có 2 sessions trên server
  3. Publish user alice

Kết quả mong đợi

✓ CẢ HAI tab Alice (Tab 1 và Tab 1b) đều nhận cùng lúc. Spring fan-out personal queue đến mọi session gắn Principal "alice".

Điểm nhấn

⚡ Đây là lý do dùng /user/queue/* thay vì tự quản sessionId: multi-device sync. User mở app trên cả desktop + mobile → đều nhận noti cùng lúc.
SCENARIO 06

In-process publish từ Spring service Java

Inject NotificationPublisher thẳng vào domain service — 0ms overhead, cùng transaction context.

Example code trong OneSubs

// OrderService.java @Service @RequiredArgsConstructor public class OrderService { private final OrderRepository orderRepo; private final NotificationPublisher publisher; @Transactional public Order createOrder(CreateOrderRequest req) { Order order = orderRepo.save(new Order(req)); // Send sau khi DB commit thành công publisher.sendToUser(order.getAssignedSaleId(), Map.of( "title", "Đơn mới #" + order.getId(), "body", order.getCustomerName(), "level", "success" )); // Broadcast cho role manager để theo dõi tổng publisher.sendToTopic("role/manager", Map.of( "title", "Đơn mới tạo", "body", "Đơn #" + order.getId() + " vừa được tạo" )); return order; } }

Điểm mạnh

✅ Zero HTTP call — publisher chỉ là method call
✅ Cùng thread + transaction — dùng TransactionSynchronization.afterCommit() để chỉ publish khi DB commit thành công
✅ Debug IDE — breakpoint trong publisher → trigger FE nhận ngay
✅ Tích hợp Spring Security — Principal trong STOMP = Principal trong SecurityContext
SCENARIO 07

STOMP Destination Model — Tổng quan

Hiểu 3 loại destination và khi nào dùng.

So sánh 3 destinations

Destination Phạm vi Route by Use case
/user/queue/* Per-user (multi-session) Principal.name Personal noti, đơn mới cho sale
/topic/* Fan-out Subscription match Role, order channel, broadcast
/app/* Client → Server @MessageMapping Chat send, client actions

Convention cho OneSubs

Loại Destination
Personal noti cho user cụ thể /user/queue/notifications
Role broadcast /topic/role/{admin|sale|tech}
Chat trên đơn hàng /topic/order/{orderId}
Emergency broadcast /topic/broadcast
Client gửi chat /app/chat/{orderId}

🎯 Tổng kết demo

Những gì đã show

  1. Embedded Tomcat — 1 port (8080) cho cả REST + WebSocket
  2. STOMP 3 destinations — /user/queue, /topic, /app
  3. Principal routing — multi-device sync tự động
  4. SimpUserRegistry — track online users không cần code
  5. In-process publish — inject bean, 0ms overhead
  6. Tích hợp Spring native — Security, MessageConverter, @MessageMapping

Vì sao OneSubs chọn phương án này

✅ Cùng stack Spring Boot — không thêm infra/process mới
✅ Same port — FE chỉ cần 1 endpoint, đơn giản nginx/CORS
✅ Two-way messaging ready — sẵn sàng cho chat Phase 2
✅ Code complexity thấp — ~5 classes, dễ maintain
✅ Scale mở rộng — có thể thay in-memory broker bằng RabbitMQ/Redis relay khi cần

Điểm cần lưu ý khi production

⚠️ Thay x-username header bằng JWT validation thực sự
⚠️ Config setAllowedOrigins thay * bằng whitelist FE URL
⚠️ Monitor SimpUserRegistry để alert khi connection leak
⚠️ Dùng @TransactionalEventListener(AFTER_COMMIT) để tránh gửi noti khi rollback
